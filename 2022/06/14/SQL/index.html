Primary Key

Combination of columns that uniquely specify a row. Have NOT NULL constraint

Foreign Key

Used to relate other tables’ primary key to join tables

Index

Performance tuning method for faster retrieval of records

Clustered Index

Reorders the physical order of the table and search based on the key values. Each table can have only one clustered index

Non Clustered Index

Do not alter the physical order of the table. Each table can have many noclustered indexes

Trigger

Stored Procedure

A function consists of many queries. Can be seen as a module or script for repeated use.

Normalization

The process of minimizing redundancy and dependency by organizing fields and tables

Denormalization

The process of introducing redundancy by incorporating data from related tables

Self Join

Useful to convert a hierarchical structure into a flat structure

Transpose a table

Performance Tuned Technique

  • The common query execution flow is to scan the whole table, will be time-consuming when data amount surge
  • Use EXPLAIN or EXPLAIN ANALYZE to see how query is executed, what steps are taken

    Index

  • Create Index
  • Ordered, need extra memory

B-Tree Index

  • Default index type
  • Used when a large of distinct values in a column (high cardinality)
  • Log(N) time to access data

Bitmap Index

  • Used when a small of distinct values in a column (low cardinality); e.g: A bool type column contains YES/NO
  • Suit for read-intensive while few writes
  • Oracle support explicit creation; Postgres not, only creates on the fly if needed
  • Fast due to bitwise operation

Hash Index

  • Only used for Equality filter (“=”), not for ranges of values
  • Smaller size than B-Tree but as fast as it
  • Used on column with unique values

Tuning Joins

  • Original Join is nested loop in essence (Full table scans)

Hash Join

  • Used only when “=” is used
  • Build on smaller table, full table scan on the bigger table

Merge Join

  • Used only when “=” is used
  • Sort the tables first then join

Partition

  • Columns will be inherited from the table being partitioned
  • Partition tables can have own indexes, constraints, triggers, etc

  • Partition by Range

  • Partition Key split by ranges, e.g: dates
  • Partition on non-overlapping keys, i.e: different partition tables cannot split by same partition key

Partition by List

  • Partition Key split by list, e.g: S&P50 stocks
  • Partition on non-overlapping keys

Partition by Hash

View

A virtual table consists of data of the subset of one or related tables. Not physically hold data, more like a projection
